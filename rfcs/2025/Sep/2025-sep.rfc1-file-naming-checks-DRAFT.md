# Glint RFC 2025-sep.rfc1: File Naming Checks

## Phase 1 — Use Case & Design Goals

### Summary

Introduce a new lint check that enforces file naming policies, mirroring the existing folder naming rules. This enables teams to apply consistent casing and allow/disallow patterns not only to folders but also to files, scoped via selectors with `kind: file` and path regexes.

### Motivation

- Today, `.glint.yaml` supports folder naming constraints but not file naming. This creates gaps in consistency and allows drift (e.g., `kebab/Case.go`).
- Many repositories enforce naming style for both folders and files; making this first-class in Glint reduces custom scripting and makes policies clearer and centrally maintained.
- The existing selector model already supports `kind: file`, so adding a file name check completes the model without breaking changes.

### Design Goals

- Match folderName semantics for files with minimal duplication.
- Use selector.kind=file and regex path scoping; no breaking config changes.
- Avoid content reads; operate on file names only (base name for predicates/prefix/suffix).
- Deterministic, concise diagnostics; reuse existing reporting severities and code patterns.
- Keep complexity low to satisfy linters (dupl/cyclomatic) via shared utilities.
- Fit existing architecture: selector decorator, factory registry, and config model.

### Backwards Compatibility

- Additive only: existing configs continue to work; keep version unchanged.
- No behavior changes to folderName; fileName is opt‑in.

### Open Questions

- Should we add optional `extensions: [".go", ".md"]` to simplify selectors? For now, rely on path regex.
- Do we expand predicates (`snake`, `pascal`) in this RFC or a follow‑up?

## Phase 2 — Design Overview, Implementation & Test Plan

### Design Overview

- New check type: `fileName`.
  - Configuration model mirrors `folderName` fields: `predicates`, `allow`, `disallow`, `prefix`, `suffix`, `prohibitPrefix`, `prohibitSuffix`, `message`, `severity`.
  - Evaluation order remains: Allow → Disallow → Predicates/Prefix/Suffix rules (same as folder check semantics).
- Selectors
  - Reuse existing `Selector` with `kind: file` and `path` regex list.
  - Example: apply to Go files under `internal/`: `selector.kind: file`, `selector.path: ["^internal/.*\\.go$"]`.
  - Prerequisite: enhance selector to actually support `kind: file`.
    - Today, `internal/app/lint/selector.go` only matches `folder`; any `file` selector never matches.
    - `internal/app/lint/requirements.go` includes a `case "file"` but it is latent/unreached until selector support is added.
    - Implementation must add `const fileKind = "file"` and update `matchesNodeKind`/`matchesKindString` accordingly.
- Reporting
  - Reuse existing reporting severities and message override.
  - IDs follow the same pattern as other checks (camel-case), e.g., `check.fileName`.
- Implementation approach
  - Add `file` checker analogous to `internal/app/lint/checks/folder`.
  - Keep a shared helper for name policy evaluation to avoid duplication; both folder and file checks call it.
  - No content reads required; only metadata and path parsing.
- Config compatibility
  - No breaking changes. Keep `version: 1`. The loader already recognizes `selector.kind: file`. We extend the dispatcher to build `fileName` checks.

### Implementation Plan

- Language constructs and locations
  - New package: `internal/app/lint/checks/file`
    - `type FileNameConfig struct { Predicates []string; Allow, Disallow []string; Prefix, Suffix, ProhibitPrefix, ProhibitSuffix, Message string }`
    - `type FileNameCheck struct { config FileNameConfig; code string; severity reporting.Severity; allowRes, disallowRes []*regexp.Regexp }`
    - `func NewFileNameCheck(cfg FileNameConfig, code string, sev reporting.Severity) lint.Checker`
    - Methods:
      - `ID() string` → "check.fileName"
      - `Apply(context.Context, *lint.Tree) []reporting.Report` → panic (node-only)
      - `ApplyToNode(context.Context, *lint.Node) []reporting.Report` → only when `n.Kind == lint.File`
      - Helpers: `allowed`, `disallowedReport`, `collectViolations`, `newReport`

  - Shared utility to minimize duplication with folder check: `internal/app/lint/util/namepolicy.go`
    - `type NameRules { Predicates []string; Allow, Disallow []string; Prefix, Suffix, ProhibitPrefix, ProhibitSuffix string }`
    - `type CompiledNameRules { allowRes, disallowRes []*regexp.Regexp; rules NameRules }`
    - `func CompileNameRules(r NameRules) CompiledNameRules`
    - `func NormalizePredicate(string) string` (maps `lower`→`lowercase`)
    - `func CheckPredicates(base string, preds []string) (ok bool, failed string)` using existing `util.CheckCasing`
    - `func Allowed(full string, res []*regexp.Regexp) bool`
    - `func FirstDisallowed(full string, res []*regexp.Regexp) (pat string, ok bool)`
    - `func Violations(base string, rules NameRules) []string` (in order: predicates, prefix, suffix, prohibitPrefix, prohibitSuffix)
    - Refactor `internal/app/lint/checks/folder/folder_name.go` to call this utility (without changing behavior).
    - Add an exported helper in `internal/app/lint/util/casing.go`: `func BaseNameNoExt(name string) string`
      - Behavior: strips the last file extension from a filename (e.g., `"a/b/MyFile.go" → "MyFile"` on just the base name), used by `fileName` predicates/prefix/suffix logic.
      - Rationale: generic utility for current and future checks; avoids scattering extension-stripping logic.

  - Selector support for files: `internal/app/lint/selector.go`
    - Add `const fileKind = "file"`
    - Update `matchesNodeKind`/`matchesKindString` to accept `file` → `NodeKind == File`.
    - Note: This unlocks the latent `case "file"` in `internal/app/lint/requirements.go`.

  - Config → plan wiring:
    - Add `type ChFileName` in `internal/app/config/model/linter.go` (same fields as `ChFolderName`).
    - Register factory constructor in `internal/app/lint/plan/factory.go`:
      - `registry["fileName"] = newFileNameCheck`
      - `func newFileNameCheck(ruleID string, severity reporting.Severity, params json.RawMessage) (lint.Checker, error)` to unmarshal `ChFileName` and build `FileNameCheck`.
    - `internal/app/lint/plan/from_config.go` already wraps checks with selectors; no change needed beyond registration.

  - Requirements/enrichment
    - `internal/app/lint/requirements.go` already flips `IncludeFiles` when selector.kind==file; no change needed. Note: the current runner includes files unconditionally (`BuildTreeFromFS(...IncludeFiles: true)`), so this optimization is currently latent.
    - No content reads required; only name/metadata. Current enrichment is sufficient.

  - Project structure alignment
    - Update `.glint.yaml`:
      - STRUCT-LINT-CHECKS: allow `^file$` in addition to `folder` and `tests` under `internal/app/lint/checks/`.
      - STRUCT-LINT-CHECKS-TESTS: allow `^file$` in addition to `folder`, `selector`, and `testutil` under `internal/app/lint/checks/tests/`.

Examples and docs
  - Add `.glint.yaml` example using both `folderName` and `fileName` side-by-side, with `selector.kind: file` scoped via path regex (e.g., `^internal/.*\.go$`).

### Test Plan

- Unit tests
  - New: `internal/app/lint/checks/tests/file/file_name_test.go`
    - Mirror folder tests: allow/disallow precedence, `kebab` and `lowercase` predicates, prefix/suffix, prohibitPrefix/prohibitSuffix, custom message, empty config.
    - Validate base-name handling (e.g., `MyComponent.go` fails `kebab`).
    - Selector scoping: `lint.Selector{Kind: "file", PathRegexes: mustCompile("^src/.+\\.go$")}`.
  - Test util: extend `internal/app/lint/checks/tests/testutil/treebuilder.go` with `File(name string)` to build file nodes beneath the current directory.
  - Add unit tests for `util.BaseNameNoExt` to ensure correct handling of multi-dot names and files without extensions.

- Integration smoke
  - Load a small config with `fileName` via plan/factory and run `lint.Lint` over an in-memory tree; assert diagnostic codes/messages.

- Coverage
  - New package(s) aim for ≥ 80% coverage; project overall remains ≥ 60%.

### Alternatives Considered

- Overloading `folderName` to handle files as well: rejected for clarity; separate `fileName` is explicit and matches selector kinds.
- Generic `name` check with `selector.kind` deciding behavior: viable, but splitting keeps user mental model straightforward and aligns with current naming.

### Risks and Mitigations

- Risk: Duplicate logic between folder and file checks.
  - Mitigation: Extract shared evaluation helpers into `internal/app/lint/util` and reuse in both checks.
- Risk: Misconfiguration due to regex path complexity.
  - Mitigation: Provide clear examples and guidance in docs; consider presets for common languages in future RFCs.

### Rollback Strategy

- The feature is additive. Rollback is a simple reversion of the new checker and dispatcher wiring.
